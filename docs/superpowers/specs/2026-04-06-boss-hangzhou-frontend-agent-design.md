# Boss 直聘杭州前端岗位采集 Agent 设计稿

日期：2026-04-06

## 1. 背景与目标

用户是前端开发，当前目标是围绕 Boss 直聘进行海投，优先解决“杭州地区有哪些公司在招前端”这一情报问题。

第一版产品定位为一个可重复运行的本地采集 Agent，而不是自动投递工具，也不是完全自主决策的浏览器智能体。它的价值是稳定记录杭州地区在招前端岗位，并按公司维度输出可直接用于海投的清单。

## 2. 目标范围

本次设计覆盖：

- 平台范围：Boss 直聘
- 地域范围：杭州
- 岗位范围：前端严格口径 + 扩展口径双层分类
- 运行方式：一次性扫描可用，架构上预留周期巡检能力
- 登录方式：用户手动登录，Agent 复用本地浏览器会话
- 数据输出：结构化本地存储 + 本地可视化看板

第一版不包含：

- 自动登录、自动过验证码、自动处理风控挑战
- 自动投递、自动打招呼、自动跟进招聘流程
- 多平台聚合
- 多城市并发采集
- 以 browser-use 或 LLM 作为主采集链路

## 3. 成功标准

第一版的成功标准是：

- 能稳定复用人工登录后的会话完成杭州前端岗位采集
- 能保留岗位事实层数据，并自动汇总出公司维度视图
- 能区分严格前端和扩展前端相关岗位
- 中断后不会整轮白跑，支持断点续跑和增量更新
- 用户可以快速回答“杭州有哪些公司在招前端”

## 4. 需求口径

### 4.1 岗位口径

使用双层口径：

- 严格口径：职位标题明确包含“前端”“Web 前端”“前端开发”等高相关关键词
- 扩展口径：职位标题或标签包含 React、Vue、H5、可视化前端、低代码前端、偏前端全栈等相关信号

系统需要同时保留两个布尔标记：

- `strict_frontend_match`
- `extended_frontend_match`

这样主库可以保持清晰，同时避免漏掉边缘但有价值的岗位。

### 4.2 结果视图

底层按岗位存全量事实，上层自动汇总成公司视图。

这意味着：

- 一个岗位对应一条岗位事实记录
- 一个公司可以关联多个岗位
- 海投使用时优先查看公司列表，再进入公司详情查看全部岗位

## 5. 方案选择

本次讨论过三种实现路径：

### 方案 A：Playwright 规则驱动

由 Playwright 负责浏览器自动化，页面解析、字段提取、去重和聚合逻辑由本地代码显式实现。

优点：

- 稳定性最好
- 最适合复用人工登录态
- 便于实现断点续跑、重试、日志和去重
- 更适合批量任务和重复运行

缺点：

- 需要手写页面规则和选择器

### 方案 B：browser-use / Agent 驱动

由高层浏览器代理根据自然语言指令自主访问页面、翻页并提取字段。

优点：

- 启动快
- 探索性任务体验好

缺点：

- 批量任务稳定性较差
- 对页面结构波动更敏感
- 在登录态、弹窗、风控、加载时机等问题上更容易漂移

### 方案 C：混合架构

主链路用 Playwright，局部难题由 Agent 或 LLM 兜底处理。

优点：

- 长期扩展性最好
- 可把 AI 能力用于疑难页面或分类补强

缺点：

- 第一版复杂度偏高
- 调试成本更高

### 推荐结论

第一版采用方案 A，后续如有需要再演进到方案 C。

原因是这次的核心目标是“稳定采集并形成公司清单”，而不是“证明自主浏览能力”。浏览器智能体可以作为补强，但不应成为 MVP 的核心依赖。

## 6. 总体架构

系统由以下模块组成：

1. 任务启动器
2. Playwright 浏览器执行器
3. 列表页采集器
4. 详情页补全器
5. 归一化与分类器
6. 存储层
7. 本地看板
8. 运行日志与失败重试模块

主流程如下：

1. 用户先手动登录 Boss 直聘
2. Agent 启动浏览器并复用已登录会话
3. 进入“杭州 + 前端相关关键词”搜索结果页
4. 翻页采集列表页摘要信息
5. 逐个进入职位详情页补齐关键字段
6. 对岗位和公司做归一化、口径分类、去重
7. 将结果增量写入本地数据库
8. 由本地看板展示公司列表和对应岗位明细

## 7. 模块设计

### 7.1 任务启动器

职责：

- 加载采集配置
- 创建本次运行的 `run_id`
- 控制页数上限、关键词集合、是否抓详情页等参数

第一版建议支持的配置项：

- 城市：固定杭州
- 关键词集合：前端严格词 + 扩展词
- 最大页数
- 是否抓详情页
- 任务速率配置

### 7.2 浏览器执行器

职责：

- 使用 Playwright 启动浏览器
- 复用用户已登录会话
- 维持稳定的页面导航、等待和截图能力

要求：

- 会话失效时能够显式报错并暂停
- 保留截图和日志，便于排障

### 7.3 列表页采集器

职责：

- 抓取每个职位卡片的摘要信息
- 提取详情页入口
- 跟踪页码和当前任务进度

列表页建议提取的字段：

- 职位标题
- 公司名
- 薪资文本
- 经验要求
- 学历要求
- 标签列表
- 工作区域
- 详情页链接或稳定标识

### 7.4 详情页补全器

职责：

- 进入职位详情页补充更完整的岗位与公司字段

详情页建议补充字段：

- 职位描述全文
- 岗位要求摘要
- 公司的行业、规模、融资阶段等可见信息
- 发布者信息中可稳定获取的字段

第一版采集到职位详情页即可，不抓公司主页。

### 7.5 归一化与分类器

职责：

- 判断是否属于严格前端或扩展前端
- 进行公司名归一
- 生成回退去重指纹

公司归一策略：

- 保留 `company_name_raw`
- 生成 `company_name_normalized`
- 初期以规则为主，例如空白、括号、常见后缀处理
- 后续可增加人工维护别名映射表

### 7.6 存储层

第一版推荐使用 SQLite 作为主存储，同时可选导出 JSON 供其他工具消费。

原因：

- 单机使用成本低
- 查询聚合方便
- 天然适合 run 记录、去重和增量更新

### 7.7 本地看板

看板的第一版目标不是复杂 BI，而是高效支持海投。

建议最小能力：

- 公司列表
- 活跃前端岗位数量
- 严格/扩展口径筛选
- 区域、薪资、经验筛选
- 查看某公司当前全部岗位及其原始链接

## 8. 数据模型

### 8.1 岗位事实表 `jobs`

建议字段：

- `job_id`
- `source_platform`
- `source_job_key`
- `source_job_url`
- `job_title`
- `company_name_raw`
- `company_name_normalized`

- `salary_text`
- `experience_text`

状态定义建议：

- `active`：本轮仍能看到
- `disappeared`：历史上见过，但当前轮次未再出现

### 8.2 公司聚合表 `companies`

建议字段：

- `size`

### 8.3 运行记录表 `runs`

建议字段：

## 9. 去重与增量更新

### 9.1 岗位级去重

优先级如下：

1. 使用站点可稳定获取的职位唯一标识
2. 如果拿不到稳定 ID，则使用稳定详情页 URL
3. 如果仍不稳定，则回退到组合指纹：
   `company_name_normalized + job_title + salary_text + district`

处理原则：

- 已存在岗位：更新 `last_seen_at` 和本轮信息
- 新岗位：插入新记录并写入 `first_seen_at`

### 9.2 公司级归一

公司视图不能直接用原始名称聚合，否则会出现简称、括号信息、空格差异导致的碎片化。

处理原则：

- 原始值永远保留
- 聚合时使用 `company_name_normalized`
- 初期允许少量误差，但必须支持后续人工映射修正

### 9.3 消失岗位判定

在一次完整扫描后：

- 本轮见到的岗位记为 `active`
- 历史存在但本轮未见到的岗位记为 `disappeared`

这样后续就能区分新增岗位和已下线岗位。

## 10. 运行机制

### 10.1 单次任务生命周期

1. 读取配置并创建 `run_id`
2. 检查登录态是否有效
3. 进入搜索页开始翻页
4. 逐页抓列表摘要
5. 进入详情页补全关键字段
6. 每处理一批即增量写库
7. 结束后生成本次采集摘要

### 10.2 断点续跑

需要记录：

- 最后成功页码
- 已抓取职位 key
- 当前 `run_id`
- 未完成详情页队列

中断后支持：

- 继续本次 run
- 或重新发起完整 run，但仍基于 `last_seen_at` 做增量更新

## 11. 异常处理

### 11.1 登录态失效

处理方式：

- 停止继续采集
- 明确提示用户重新手动登录
- 保留当前 run 日志和进度

### 11.2 页面结构变化

处理方式：

- 记录失败选择器
- 截图保存现场
- 将当前页面记入失败清单
- 不因单个失败阻塞整个任务

### 11.3 详情页偶发失败

处理方式：

- 先重试有限次数
- 仍失败则进入重试队列
- 在任务尾部统一补扫

### 11.4 疑似风控

表现可能包括：

- 页面加载异常
- 访问频率限制
- 人机校验出现

处理方式：

- 主动降低速率
- 必要时结束任务
- 不尝试自动绕过验证

## 12. 测试与验证策略

第一版至少覆盖以下验证层次：

### 12.1 解析规则测试

- 选择器能否稳定提取列表字段
- 选择器能否稳定提取详情字段

### 12.2 去重与归一测试

- 同一职位二次出现不会重复入库
- 同一公司不同名称形式可正确聚合

### 12.3 运行时容错测试

- 中途终止后能否断点续跑
- 某个详情页失败时主流程是否仍继续

### 12.4 看板验证

- 公司视图数量是否符合预期
- 某公司下是否能看到完整岗位明细
- 严格/扩展筛选是否生效

## 13. 风险与边界

本设计默认用户已自行处理账号登录，并接受需要人工接管登录失效和验证挑战的情况。

设计重点是信息采集和本地整理，不包含自动规避站点限制的能力。第一版应把速率、重试、日志和可中断恢复放在高优先级，以换取稳定性和可维护性。

## 14. 第一版交付物

第一版应产出：

- 一个本地运行的 Playwright 采集 Agent
- 一个 SQLite 数据库
- 一个可选 JSON 导出
- 一个本地网页看板
- 一组运行日志与失败记录

## 15. 后续演进方向

在第一版稳定后，可以按以下顺序扩展：

1. 增加周期巡检能力
2. 增加人工校正公司别名映射
3. 增加 AI 辅助分类与疑难页面抽取
4. 扩展到其他城市
5. 扩展到其他招聘平台

## 16. 最终结论

这不是一个以“智能体自主浏览”为核心卖点的产品，而是一个以“稳定岗位情报采集”为核心目标的工具。

第一版最合适的设计是：

- 以 Playwright 为主采集内核
- 复用人工登录态
- 抓取列表页和详情页
- 底层存岗位事实，顶层做公司聚合
- 使用 SQLite 作为主存储
- 提供本地看板支撑海投

这个方案最符合当前用户目标，也为后续增加周期巡检和 AI 补强保留了空间。
